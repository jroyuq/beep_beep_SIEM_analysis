Ce projet a pour objectif de démontrer comment analyser des logs de sécurité en utilisant une approche de traitement de flux inspirée des systèmes SIEM. L’idée principale est de transformer un simple fichier de logs Linux en un flux d’événements analysés en temps réel afin de détecter des comportements suspects comme des attaques brute force, des scans d’utilisateurs ou des actions sensibles exécutées avec des privilèges élevés. Le projet s’inscrit dans un contexte de cybersécurité où la détection rapide d’anomalies est essentielle pour protéger un système, et il s’appuie sur une logique similaire à celle des outils professionnels comme Wazuh ou Splunk.

Le projet repose sur deux éléments principaux : un fichier de logs appelé auth.log et un programme Java nommé MonitorWazuhAuthLog. Le fichier auth.log contient des événements d’authentification provenant d’un système Linux, comme des connexions SSH, des erreurs d’authentification ou des commandes exécutées avec sudo. Ce type de fichier est généralement disponible sur les systèmes Linux dans /var/log/auth.log. Pour tester le projet, il est possible d’utiliser un log réel ou de créer un fichier simulé. Par exemple, sous Linux, on peut copier un vrai log avec la commande cp /var/log/auth.log ./auth.log, ou générer des événements de test avec des connexions SSH répétées.

Le fonctionnement du projet repose sur l’utilisation de BeepBeep, un moteur de traitement de flux permettant de lire les données ligne par ligne. Contrairement à une approche classique où un fichier est chargé entièrement en mémoire, BeepBeep permet de traiter les données en continu, événement par événement. La bibliothèque BeepBeep peut être téléchargée depuis son dépôt officiel à l’adresse suivante : https://github.com/liflab/beepbeep-3
. Le fichier .jar doit ensuite être ajouté au projet pour permettre la compilation et l’exécution du programme.

Pour installer et exécuter le projet, il est nécessaire d’avoir Java installé sur la machine. On peut vérifier la version avec la commande java -version. Si Java n’est pas installé, il est recommandé d’installer une version récente du JDK depuis https://adoptium.net
. Une fois Java installé, le projet doit être organisé avec le fichier auth.log et le fichier Java dans le même dossier, ainsi que le fichier beepbeep-3.13.jar. La compilation peut être réalisée avec la commande javac -cp .;beepbeep-3.13.jar MonitorWazuhAuthLog.java sous Windows, ou javac -cp .:beepbeep-3.13.jar MonitorWazuhAuthLog.java sous Linux. L’exécution se fait ensuite avec java -cp .;beepbeep-3.13.jar MonitorWazuhAuthLog.

Lors de l’exécution, le programme utilise le composant ReadLines de BeepBeep pour lire le fichier auth.log. Ce composant transforme le fichier en flux d’événements, où chaque ligne correspond à un événement distinct. Le programme récupère ensuite ces événements un par un grâce à un objet Pullable, ce qui permet d’appliquer les règles de détection en temps réel. Cette approche permet de simuler un système de surveillance continu, comme dans un SIEM réel.

La première règle implémentée concerne la détection de brute force SSH. Le programme analyse les lignes contenant le message “maximum authentication attempts exceeded” et incrémente un compteur pour chaque adresse IP. Si le nombre d’échecs dépasse un certain seuil, une alerte critique est générée. Cette logique permet de détecter des attaques automatisées visant à compromettre un compte via des tentatives répétées.

La deuxième règle permet de détecter des tentatives de connexion sur des utilisateurs inexistants. Le programme identifie les messages “Invalid user” et compte le nombre de tentatives par adresse IP. Lorsqu’un seuil est atteint, une alerte est générée pour signaler un comportement suspect correspondant à une phase de reconnaissance d’un attaquant.

La troisième règle concerne les connexions réussies. Le programme détecte les événements “Accepted publickey” et enregistre l’utilisateur ainsi que l’adresse IP. Cette information permet de suivre les accès légitimes et d’identifier une activité suspecte après une connexion valide.

La quatrième règle permet de détecter l’exécution de commandes sudo sensibles. Le programme analyse les commandes exécutées avec des privilèges élevés et vérifie si elles correspondent à des actions potentiellement dangereuses comme l’utilisation de curl, wget ou apt-get install. Si une commande suspecte est détectée, une alerte est générée. Cette règle permet d’identifier des comportements malveillants après une compromission.

L’architecture du projet suit un modèle simple mais représentatif d’un SIEM. Le fichier de logs constitue la source des données, BeepBeep agit comme un moteur de transformation qui convertit ces données en flux d’événements, le programme Java applique des règles de détection, et les résultats sont affichés sous forme d’alertes. Cette chaîne correspond aux étapes classiques d’un SIEM : ingestion, transformation, analyse et détection.

Ce projet met en évidence l’intérêt du traitement de flux en cybersécurité. Plutôt que d’analyser les logs après coup avec des outils comme grep ou cat, il permet de détecter des attaques dès leur apparition. Par exemple, une recherche classique pourrait être faite avec grep "Invalid user" auth.log, mais cette approche reste statique. À l’inverse, l’approche utilisée ici permet une détection dynamique et continue.

En conclusion, ce projet constitue une démonstration concrète de l’utilisation de BeepBeep pour analyser des logs de sécurité en temps réel. Il montre comment transformer des données brutes en informations exploitables et comment construire un système de détection proche de ceux utilisés dans des environnements professionnels.
